# 1 ИССЛЕДОВАНИЕ ПРЕДМЕТНОЙ ОБЛАСТИ

## 1.1 Описание предметной области

### 1.1.1 Виртуальные лаборатории в образовании

Виртуальные лаборатории как класс образовательного программного обеспечения возникли на рубеже 2000-х годов как ответ на потребность в массовой доступности практических учебных заданий. Традиционный подход предполагал наличие физических компьютерных классов с предустановленным специализированным программным обеспечением — компиляторами, интерпретаторами, системами управления базами данных. Такой подход был сопряжён с высокой ресурсоёмкостью администрирования и пространственными ограничениями.

С распространением широкополосного интернета появились первые серверные интерактивные среды: пользователь вводит код в браузере, код отправляется на сервер, выполняется там, и результат возвращается обратно. Этот подход быстро стал доминирующим: CodeChef (2009), Codeforces (2010), Stepik (2013), Replit (2016) — все используют именно серверное исполнение.

В области естественных наук развивался параллельный класс платформ — симуляторы физических и химических процессов: PhET Interactive Simulations [18], Labster (2012). Эти платформы изначально ориентированы на визуализацию и симуляцию, а не на исполнение пользовательского кода, и традиционно реализуются на стороне клиента (JavaScript или Adobe Flash).

Для дисциплин, связанных с программированием, ситуация кардинально иная: необходимо исполнять произвольный код, написанный обучающимся, в изолированной среде с контролем ресурсов. До 2019 года единственным надёжным способом обеспечить такую изоляцию были серверные контейнеры (Docker, LXC).

### 1.1.2 Проблемы серверного подхода

**Задержка отклика.** Исполнение кода на сервере включает как минимум одно полное сетевое путешествие запроса: от браузера обучающегося до сервера и обратно. Измеренное среднее RTT от клиента до типичного облачного датацентра составляет 40–80 мс при хорошем соединении [1]. К этому добавляется накладной расход самого Docker-контейнера: при «тёплом» старте — 100–150 мс на подготовку контекста, при «холодном» старте (container cold start) — 350–500 мс. Итоговая задержка при каждом запуске кода составляет в среднем 150–200 мс при тёплом и 450–600 мс при холодном контейнере.

Исследования в области когнитивной психологии [2] показывают, что задержка свыше 100 мс разрушает ощущение непрерывности взаимодействия с системой, что критически важно для итеративного процесса написания и отладки кода.

**Стоимость масштабирования.** Серверная модель исполнения кода линейно масштабирует потребление ресурсов с ростом числа одновременных пользователей. Каждый активно работающий студент требует выделенной вычислительной мощности: обычно 0,5–1 vCPU и 256–512 МБ RAM для изолированного Python-контейнера. При 1000 одновременных пользователях необходим кластер из 3–5 виртуальных машин класса 4 vCPU / 16 GB RAM с запасом для пиковой нагрузки.

**Зависимость от сети.** При отсутствии или нестабильности интернет-соединения серверные платформы полностью неработоспособны. Это актуально для учебных аудиторий с перегруженной Wi-Fi-сетью, для обучающихся в удалённых регионах, а также для сценариев работы вне кампуса.

**Отсутствие совместной работы.** Большинство серверных платформ (Stepik, Codeforces) не поддерживают синхронное совместное редактирование кода несколькими участниками. Реализация такой функциональности на серверной стороне значительно усложняет архитектуру и увеличивает аппаратную ресурсоёмкость, что подтверждается работами по совместному программированию в учебной среде [7].

### 1.1.3 Тренд на Edge Computing и Local-First

Edge Computing — парадигма, перемещающая вычисления ближе к источнику данных и пользователю. В контексте образовательных платформ «краем сети» является непосредственно браузер обучающегося — устройство, уже содержащее процессор, память и хранилище.

Концепция Local-First [5] формализует этот подход применительно к прикладным программам: данные хранятся локально, все операции выполняются без задержки, а синхронизация происходит асинхронно в фоне. Local-First системы обеспечивают высокую отзывчивость, офлайн-доступность и устойчивость к сетевым сбоям.

Стандартизация WebAssembly консорциумом W3C в 2019 году [13] открыла новые возможности для реализации вычислительно ёмких приложений в браузере. WASM позволяет компилировать код на C/C++/Rust в портативный бинарный формат, выполняемый в изолированной виртуальной машине браузера с производительностью, в 10–20 раз превышающей интерпретируемый JavaScript [3, 8].

## 1.2 Обзор и сравнение существующих решений

### 1.2.1 Анализ существующих платформ виртуальных лабораторий

**Stepik.** Stepik — российская платформа для создания интерактивных онлайн-курсов (MOOC), основанная в 2013 году. Предоставляет встроенную проверку кода для Python, C++, Java и ряда других языков.

Архитектура исполнения серверная: код студента выполняется в Docker-контейнерах на стороне Stepik. Для каждой попытки запуска создаётся изолированный контейнер с ограничениями по CPU (0,5 vCPU) и памяти (256 МБ). Совместная работа отсутствует: каждый студент работает в своей независимой сессии. Офлайн-режим отсутствует. Преимущества — богатая экосистема курсов, поддержка видеолекций, система аналитики прогресса, интеграция с сертификацией. Ограничения — задержка при исполнении кода 200–500 мс, отсутствие совместной работы, полная зависимость от серверной инфраструктуры.

**Replit.** Replit — американская платформа (2016) для совместной разработки в браузере, поддерживающая более 50 языков программирования.

Архитектура исполнения серверная: каждый проект запускается в отдельной виртуальной машине на базе nsjail/Firecracker. Совместная работа реализована через собственный Operational Transformation протокол (Multiplayer mode). Офлайн-режим частичный: редактирование кода доступно, исполнение — нет. Преимущества — полноценная IDE в браузере, совместная работа, поддержка баз данных и внешних зависимостей. Ограничения — платная подписка для командной работы, высокие задержки при «холодном» старте VM (5–15 с), данные хранятся исключительно на серверах компании, что создаёт риски конфиденциальности.

**JupyterHub.** JupyterHub — серверная платформа (Project Jupyter, 2015) для многопользовательских окружений Jupyter Notebook, широко применяемая в академической среде для курсов по Data Science и научным вычислениям [9].

Архитектура исполнения серверная: каждый пользователь получает изолированный Jupyter-сервер в Docker-контейнере или Kubernetes Pod. Совместная работа ограниченная: через расширение JupyterLab-rtc реализовано совместное редактирование notebook, однако требует дополнительной настройки. Офлайн-режим отсутствует. Преимущества — богатая экосистема расширений, нативная поддержка научных библиотек (NumPy, Pandas, Matplotlib), возможность self-hosting. Ограничения — сложная конфигурация и поддержка, высокие требования к серверной инфраструктуре, не предназначен для образовательных заданий с тест-кейсами.

**CodeSandbox.** CodeSandbox — онлайн-IDE (2017) для веб-разработки, специализирующаяся на React, Vue, Angular и аналогичных фреймворках.

Архитектура исполнения гибридная: статические проекты выполняются в браузере (через Sandpack — WASM-сборщик), серверные — в облаке. Совместная работа реализована через собственный протокол синхронизации (Live Sessions). Офлайн-режим частичный (только Sandpack-проекты). Преимущества — отличная поддержка экосистемы веб-разработки, интеграция с GitHub/npm. Ограничения — не предназначен для образовательного контента с заданиями и тест-кейсами, ориентирован исключительно на веб-технологии.

**PhET Interactive Simulations.** PhET (Physics Education Technology, University of Colorado, 2002) — библиотека интерактивных симуляций для физики, химии, математики и биологии [18].

Архитектура исполнения клиентская: все симуляции написаны на JavaScript/TypeScript и выполняются непосредственно в браузере. Совместная работа отсутствует. Офлайн-режим полный: симуляции доступны в виде автономных HTML-файлов. Преимущества — отсутствие нагрузки на серверную инфраструктуру, полная офлайн-доступность, богатая библиотека. Ограничения — не поддерживает исполнение пользовательского кода; является симулятором, а не программируемой средой.

**Pyodide Sandbox.** Pyodide [17] — порт интерпретатора CPython 3.11 в WebAssembly. Позволяет выполнять Python-код непосредственно в браузере.

Архитектура исполнения клиентская (WASM). Совместная работа отсутствует (является библиотекой, не платформой). Офлайн-режим возможен после загрузки WASM-бинарника (около 10 МБ). Преимущества — производительность, близкая к нативному Python; поддержка NumPy, Pandas и других C-расширений. Ограничения — является низкоуровневой библиотекой без пользовательского интерфейса и образовательной оболочки.

### 1.2.2 Сравнение технологий исполнения кода в браузере

Исторически первым подходом к исполнению нестандартных языков в браузере было написание интерпретаторов на JavaScript. Такой подход прост в интеграции, но ограничен производительностью интерпретируемого JS-кода.

**JavaScript-интерпретаторы.** Skulpt — реализация Python 2/3 на JavaScript. Выполняет Python-код в нативном JS-движке браузера без WASM. Преимущества: малый размер загрузки (около 1 МБ), простота интеграции. Ограничения: производительность в 10–50 раз хуже CPython, отсутствие поддержки C-расширений (NumPy, Pandas). Brython — альтернативная JS-реализация Python, ориентированная на взаимодействие с DOM. Не предназначена для вычислительно ёмкого кода.

**WebAssembly-движки.** Pyodide — CPython 3.11, скомпилированный через Emscripten в WASM [17]. Поддерживает полный стандарт Python и многие C-расширения (NumPy, Pandas, Matplotlib, SciPy). Размер бинарника: около 10 МБ base + зависимости. Производительность: 80–90 % от нативного CPython.

Wasmoon [21] — реализация Lua 5.4 VM в WebAssembly с нативным WASM-ядром. Компактный размер (около 400 КБ), быстрая инициализация (около 150 мс).

sql.js [22] — SQLite, скомпилированный в WebAssembly. Позволяет выполнять SQL-запросы в браузере с полной совместимостью с SQLite 3.

Нативный JS-движок (V8) — для JavaScript используется встроенный движок браузера; никакой дополнительной загрузки не требуется, время инициализации минимально.

### 1.2.3 Сравнение подходов к синхронизации данных

Для реализации совместного редактирования кода в реальном времени необходим механизм синхронизации, гарантирующий согласованность состояния у всех участников при конкурентных изменениях.

**Operational Transformation (OT).** Operational Transformation — подход, разработанный Эллисом и Гиббсом в 1989 году [15] и лежащий в основе Google Docs. Принцип: каждое изменение представляется как операция (вставка/удаление символов на позиции N). При конкурентных операциях каждый узел трансформирует входящие операции относительно уже применённых локальных операций.

Формально, OT требует выполнения двух условий корректности — TP1 и TP2 [10]. Условие TP1 гарантирует эквивалентность результата при различном порядке применения двух операций. Условие TP2 распространяет это свойство на тройки операций. В работе [14] доказано, что для систем с N > 2 участниками выполнение TP2 без центрального сервера-арбитра, устанавливающего тотальный порядок операций, принципиально не гарантируется. Именно поэтому все промышленные OT-системы (Google Docs, Etherpad) используют выделенный сервер для сериализации конкурентных операций.

Ключевое ограничение для Local-First: OT требует постоянной связи с сервером-арбитром. При отсутствии сетевого соединения редактирование либо блокируется полностью, либо при последующей синхронизации возникают неразрешимые конфликты. Это фундаментально несовместимо с принципом Local-First.

**CRDT (Conflict-free Replicated Data Types).** CRDT — семейство структур данных, введённых в работе [4], математически гарантирующих конечную согласованность (Strong Eventual Consistency) без центрального координатора. В отличие от OT, CRDT не требуют тотального упорядочивания операций: вместо этого каждая операция разработана так, чтобы её применение было коммутативным, ассоциативным и идемпотентным.

CRDT-системы разделяются на два класса: state-based (CvRDT) и operation-based (CmRDT). В state-based системах узлы периодически обмениваются полным состоянием, объединяемым через функцию слияния `merge`. В operation-based системах передаются только операции изменения. Yjs реализует гибридный подход: для первоначальной синхронизации используется полный снапшот состояния (state-based), для инкрементальной — операционные дельты (operation-based) [12].

Для текстового совместного редактирования применяются Sequence CRDT, к которым относится алгоритм YATA [6], реализованный в Yjs `Y.Text`. В работе [16] продемонстрировано, что CRDT успешно применяются для rich-text редактирования с форматированием в production-системах, что подтверждает зрелость подхода.

Преимущества CRDT перед OT:
- не требуется центральный арбитр — корректно работают в P2P и офлайн-режимах;
- конечная согласованность математически доказана для любого порядка доставки операций;
- горизонтальное масштабирование без усложнения серверной логики.

Ограничения CRDT:
- более высокое потребление памяти (хранение идентификаторов и tombstone-записей);
- удалённые элементы не удаляются физически (tombstone) — при интенсивном редактировании документ растёт в памяти;
- реализация алгоритма сложнее, чем у простого OT для двух участников.

**Yjs.** Yjs [12] — CRDT-библиотека, реализующая несколько типов общих данных: `Y.Text`, `Y.Array`, `Y.Map`, `Y.XmlFragment`. Поддерживает множество провайдеров синхронизации: WebRTC (y-webrtc), WebSocket (y-websocket), IndexedDB (y-indexeddb).

Для данной работы Yjs является оптимальным выбором: зрелая библиотека с активной поддержкой, официальный биндинг для Monaco Editor (y-monaco) [19], подтверждённая производительность при больших документах, и — что критически важно — поддержка офлайн-персистентности через `Y.encodeStateAsUpdate` / `Y.applyUpdate`.

### 1.2.4 Математические основы CRDT и алгоритм YATA

Данный подраздел содержит формальное описание математической основы CRDT и конкретного алгоритма синхронизации текста, применённого в разрабатываемой системе.

**Формальное определение CRDT.**

Определение полурешётки объединения (join-semilattice). Упорядоченное множество (S, ≤) является полурешёткой объединения, если для любых двух элементов s, t ∈ S существует наименьшая верхняя грань (supremum) s ⊔ t ∈ S. Оператор ⊔ (join, слияние) обладает следующими алгебраическими свойствами:

- коммутативность: s ⊔ t = t ⊔ s;
- ассоциативность: (s ⊔ t) ⊔ u = s ⊔ (t ⊔ u);
- идемпотентность: s ⊔ s = s.

Определение state-based CRDT (CvRDT). Реплицированный тип данных (S, s₀, q, u, m) является CvRDT, если (S, ≤) — полурешётка объединения, где S — множество состояний, s₀ ∈ S — начальное состояние, q: S → V — функция запроса, u: S → S — функция обновления, монотонная: u(s) ≥ s, m: S × S → S — функция слияния: m(s, t) = s ⊔ t.

Теорема о сходимости [4]: если два узла i и j получили одинаковые множества обновлений (в любом порядке), то их состояния совпадают: q(sᵢ) = q(sⱼ). Это свойство называется Strong Eventual Consistency.

Доказательство следует непосредственно из свойств полурешётки: поскольку слияние коммутативно и ассоциативно, порядок применения операций не влияет на итоговое состояние. Идемпотентность гарантирует, что повторное применение одного и того же обновления (например, при ретрансляции через SignalR) не изменяет состояние.

**Алгоритм YATA.**

YATA (Yet Another Transformation Approach) — алгоритм CRDT для последовательностей, разработанный в работе [6] специально для совместного редактирования текста. Именно на YATA основан тип `Y.Text` в библиотеке Yjs.

Структура элемента последовательности. Каждый вставленный символ (или блок символов) представлен структурой:

```
Item {
  id:      (clientID: uint64, clock: uint64),  // глобально уникальный идентификатор
  content: string,                              // содержимое (один или несколько символов)
  left:    Item | null,                         // левый сосед в момент вставки
  right:   Item | null,                         // правый сосед в момент вставки
  deleted: bool                                 // tombstone-флаг
}
```

Идентификатор (clientID, clock) уникален глобально: clientID назначается случайным образом при первом запуске клиента, clock — монотонно возрастающий счётчик операций данного клиента (логические часы Лэмпорта).

Операция вставки: Insert(id, content, leftOrigin, rightOrigin) — вставить content в позицию между leftOrigin и rightOrigin. После применения операции новый элемент становится частью связного списка элементов документа.

Разрешение конфликтов при одновременной вставке. При одновременной вставке в одну позицию двумя клиентами A и B возникает конфликт: оба указывают одинаковые leftOrigin и rightOrigin. YATA разрешает его детерминированно через сравнение clientID: если left(A) = left(B) и right(A) = right(B), элементы упорядочиваются по убыванию clientID — clientID(A) > clientID(B) приводит к вставке A перед B; clientID(B) > clientID(A) — наоборот.

Поскольку сравнение clientID детерминировано и одинаково на всех репликах, все узлы придут к одному и тому же порядку элементов, независимо от порядка получения операций. Это обеспечивает Strong Eventual Consistency.

Операция удаления. Вместо физического удаления элемента устанавливается флаг deleted = true (tombstone). Это обеспечивает идемпотентность: повторное удаление уже удалённого элемента не изменяет состояние. Tombstone-элементы остаются в структуре данных, сохраняя контекст для корректного разрешения конфликтов вставок в окрестности удалённых позиций.

Сложность операций. Применение одной операции вставки имеет сложность O(D), где D — число конкурентных операций, вставленных в ту же позицию. В типичном учебном сценарии (2–5 участников) D мало, поэтому практическая производительность близка к O(1). Наивная реализация sequence CRDT имеет сложность O(N) по числу всех элементов документа; YATA оптимизирует это до O(D) за счёт использования ссылок left/right вместо линейного поиска позиции.

**Применение YATA в разрабатываемой системе.**

В системе используется `Y.Map<filePath, Y.Text>`, где каждый `Y.Text` представляет отдельный файл лабораторной работы и содержит YATA-последовательность символов. `Y.Map` реализован как LWW-Map (Last-Write-Wins): конфликт при одновременном создании файла с одинаковым путём разрешается в пользу операции с большим clock.

Снапшоты состояния `Y.Doc` кодируются через `Y.encodeStateAsUpdate` в компактный бинарный формат (структурированный YATA-граф с дельта-сжатием) и декодируются через `Y.applyUpdate`. Это позволяет эффективно хранить полное состояние в localStorage и передавать дельты через SignalR в MessagePack-кодировке.

## 1.3 Определение требований

На основании проведённого анализа предметной области, существующих решений и технологий сформулированы требования к разрабатываемой системе.

Диаграмма вариантов использования, наглядно отражающая взаимодействие акторов с функциями системы, приведена в **Приложении Б.1**.

### 1.3.1 Функциональные требования

ФТ-1. Система должна обеспечивать исполнение пользовательского кода на языках Python, JavaScript, Lua и SQL непосредственно в браузере без обращения к серверу.

ФТ-2. Система должна поддерживать многофайловые проекты в рамках одной лабораторной работы с возможностью указания главного файла для исполнения.

ФТ-3. Система должна предоставлять функциональность создания и редактирования курсов, содержащих упорядоченные лабораторные работы.

ФТ-4. Каждая лабораторная работа должна поддерживать набор тест-кейсов (пар «ввод stdin» — «ожидаемый вывод stdout») для автоматической проверки кода обучающегося.

ФТ-5. Система должна обеспечивать совместное редактирование кода несколькими участниками в реальном времени с отображением списка участников сессии.

ФТ-6. Система должна функционировать в офлайн-режиме после первоначальной загрузки приложения и WASM-движков.

ФТ-7. Система должна обеспечивать экспорт курса в формат JSON и импорт из JSON-файла.

ФТ-8. Система должна включать встроенный профилировщик производительности WASM-исполнения с отображением статистики (минимум, максимум, среднее, медиана, 95-й перцентиль) и экспортом отчёта.

ФТ-9. Пользователь должен иметь возможность сбросить код лабораторной работы к исходному шаблону.

ФТ-10. Система должна обеспечивать защиту определённых файлов лабораторной работы от редактирования обучающимися (read-only-файлы).

ФТ-11. Система должна поддерживать ролевое распределение в совместных лабораторных работах с разграничением доступа к файлам по ролям.

ФТ-12. Система должна поддерживать сравнительный режим профилировщика с одновременным запуском кода в WASM-среде и на сервере для прямого сопоставления времени выполнения.

### 1.3.2 Нефункциональные требования

НФТ-1. Производительность. Время исполнения эталонного кода (FizzBuzz N=100, Python) после инициализации движка не должно превышать 50 мс в медиане.

НФТ-2. Изоляция. Исполнение кода с бесконечным циклом или избыточным потреблением памяти должно прерываться принудительным завершением Web Worker не позднее чем через 5 секунд.

НФТ-3. Офлайн-доступность. После первоначального кэширования приложение должно полностью запускаться и функционировать (в части создания и редактирования кода) без подключения к сети.

НФТ-4. Установка как PWA. Приложение должно соответствовать критериям Progressive Web Application и допускать установку на устройство через браузерный механизм.

НФТ-5. Масштабируемость серверной части. Серверный компонент в режиме CRDT-ретрансляции должен быть спроектирован как stateless-ретранслятор, не хранящий состояния кода обучающихся, что обеспечивает горизонтальное масштабирование без дополнительной сложности.

НФТ-6. Безопасность. Пользовательский код не должен иметь доступа к файловой системе устройства, DOM родительской страницы или другим вкладкам браузера; изоляция обеспечивается WebAssembly-песочницей и Web Worker.

НФТ-7. Кросс-браузерная совместимость. Приложение должно корректно функционировать в современных версиях браузеров на движках V8 (Chromium, Edge), SpiderMonkey (Firefox) и JavaScriptCore (Safari).

НФТ-8. Объём загрузки. Размер первоначальной загрузки оболочки приложения (без WASM-движков) не должен превышать 2 МБ; WASM-движки загружаются по запросу.
