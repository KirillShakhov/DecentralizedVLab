# 2 ПРОЕКТИРОВАНИЕ СИСТЕМЫ

## 2.1 Технологии и инструменты разработки

На основании требований, сформулированных в главе 1, выполнен выбор технологического стека. Ключевым критерием при выборе каждого инструмента являлась совместимость с Local-First архитектурой: приоритет отдавался решениям, способным функционировать без постоянного серверного соединения.

### 2.1.1 Фронтенд

**TypeScript и React (Vite).** TypeScript — статически типизированное надмножество JavaScript. Использование статической типизации в данном проекте критически важно: интерфейсы типов `Course`, `Lab`, `TestCase`, `User`, `Participant` формируют контракт между клиентской и серверной частями, минимизируя класс ошибок несоответствия структур данных при сериализации/десериализации.

React 19 выбран как доминирующий фреймворк для сложных интерактивных одностраничных приложений с развитой экосистемой. Компонентная модель React хорошо соответствует архитектуре рабочего пространства: независимые компоненты `FileTree`, `MultiFileEditor`, `Terminal`, `TestPanel`, `ProfilerPanel` монтируются и размонтируются без взаимных зависимостей.

Vite 7 выбран как инструмент сборки вместо Create React App по причине значительно более быстрого времени сборки (HMR менее 100 мс) и нативной поддержки ES-модулей без дополнительной конфигурации. Дополнительно используется плагин `vite-plugin-pwa` для генерации Service Worker и манифеста PWA.

**Monaco Editor.** Monaco Editor — движок редактора кода, лежащий в основе Visual Studio Code [19]. Выбор обусловлен несколькими факторами. Во-первых, Monaco обеспечивает профессиональный пользовательский опыт редактирования кода: подсветка синтаксиса для 40+ языков, IntelliSense, мультикурсорное редактирование. Во-вторых, Monaco предоставляет низкоуровневый API управления моделями документов (`ITextModel`), что необходимо для реализации многофайловых вкладок без пересоздания экземпляра редактора. В-третьих, для Monaco существует официальный CRDT-биндинг `y-monaco`, реализующий двустороннюю синхронизацию между `Y.Text` и Monaco `ITextModel`.

Альтернатива — CodeMirror 6 — рассматривалась, однако экосистема `y-codemirror` менее зрелая по сравнению с `y-monaco`, а API `ITextModel` Monaco удобнее для многофайловой архитектуры.

**Material UI (MUI v7).** Material UI — компонентная библиотека, реализующая Material Design. Использование готовой системы компонентов (`Box`, `Paper`, `Select`, `Chip`, `IconButton`, `Collapse`) позволило сосредоточиться на бизнес-логике приложения, а не на разработке UI-примитивов. Тёмная тема настроена как глобальная для создания профессионального вида IDE.

**Yjs и y-monaco.** Yjs — CRDT-библиотека [12], обоснованная в подразделе 1.2.3. Структура данных `Y.Map<filePath, Y.Text>` позволяет хранить все файлы лабораторной работы в одном `Y.Doc` с автоматической синхронизацией. `y-monaco` реализует `MonacoBinding` — объект, связывающий `Y.Text` с `ITextModel` и обрабатывающий трансляцию операций в обе стороны.

**@microsoft/signalr и @microsoft/signalr-protocol-msgpack.** Клиентская часть SignalR обеспечивает подключение к серверному хабу с автоматическим переподключением [11]. MessagePack-протокол выбран вместо JSON по причине меньшего размера сообщений (бинарный формат): CRDT-дельты представляют собой массивы байт, которые в MessagePack передаются в нативном бинарном виде, тогда как JSON требует base64-кодирования.

### 2.1.2 WASM-движки

**Pyodide 0.25.** Pyodide — полная реализация CPython 3.11 в WebAssembly, разработанная Mozilla и переданная в Pyodide community [17]. Включает менеджер пакетов micropip, поддерживающий установку Python-пакетов прямо в браузере. Для данной работы используется файловая система Emscripten (MEMFS) для монтирования многофайловых проектов: каждый файл лабораторной работы записывается в виртуальную файловую систему перед запуском скрипта.

**Wasmoon 1.16.** Wasmoon — реализация Lua 5.4 VM в WebAssembly с нативным WASM-ядром [21]. Выбран за компактность (около 400 КБ), быструю инициализацию (около 150 мс) и поддержку актуальной версии языка Lua. В первоначальном варианте проекта рассматривалась альтернатива Fengari (Lua 5.3 VM на чистом JavaScript), однако Wasmoon обеспечивает заметно лучшую производительность за счёт выполнения VM в WASM.

**sql.js.** sql.js — SQLite 3, скомпилированный в WebAssembly [22]. Предоставляет полный SQL-интерфейс в браузере. База данных создаётся в памяти при каждом запуске лабораторной работы, что соответствует модели «исполнение → вывод результата» без сохранения состояния между запусками.

**Нативный V8.** Для исполнения JavaScript используется встроенный движок браузера. Дополнительная WASM-загрузка не требуется, время инициализации минимально. Изоляция исполнения обеспечивается через `new Function(code)` с перехватом `console.log`/`error`/`warn`.

### 2.1.3 Бэкенд

**C# ASP.NET Core 8 и SignalR.** Серверная часть реализована на C# ASP.NET Core 8. Выбор C# вместо Node.js или Java обусловлен зрелостью реализации SignalR в экосистеме .NET: библиотека разработана Microsoft [11], обеспечивает нативную интеграцию с ASP.NET Core middleware, поддерживает горизонтальное масштабирование через Redis Backplane.

SignalR Hub реализует минимальный набор операций совместной работы: `JoinRoom`, `LeaveRoom`, `SendDocumentUpdate`. В режиме совместной работы сервер является stateless-ретранслятором: он не хранит содержимое кода и не интерпретирует CRDT-дельты — только пересылает их участникам комнаты.

Помимо CRDT-ретранслятора, серверная часть включает:

- REST-контроллер `CourseSyncController` для синхронизации курсов между устройствами одного преподавателя или между преподавателем и обучающимися. Курсы хранятся в SQLite через Entity Framework Core с моделью «JSON-документ + временная метка». Применяется стратегия Last-Write-Wins по полю `updatedAt`. Это даёт преподавателю возможность редактировать курс на одном устройстве и распространять обновления, не нарушая Local-First-принцип для исполняемого кода.
- REST-контроллер `ExecuteController` совместно с сервисом `CodeRunner`, выполняющим серверное исполнение учебного кода. Этот эндпоинт используется исключительно профилировщиком в режиме сравнения для прямого замера серверного времени и не задействуется в основном потоке работы обучающегося.

**API Gateway (YARP).** Дополнительный сервис на основе YARP (Yet Another Reverse Proxy) выполняет роль единой точки входа: проксирует HTTP-запросы к ASP.NET Core API и WebSocket-соединения SignalR, обеспечивает корректную обработку `X-Forwarded-For` заголовков.

**Docker Compose и nginx.** Развёртывание осуществляется через Docker Compose: контейнер nginx обрабатывает HTTP-запросы и проксирует WebSocket-соединения SignalR на ASP.NET Core контейнер. Nginx также отдаёт статические файлы фронтенда (собранный Vite-бандл) и WASM-бинарники.

### 2.1.4 Хранение данных

**IndexedDB.** IndexedDB — основное хранилище данных на клиентской стороне. Используется для хранения курсов (хранилище `courses`) и метаданных сессий (хранилище `sessions`). Доступ реализован через собственный типизированный wrapper на базе нативного IDB API, предоставляющий функции `get<T>`, `getAll<T>`, `save`, `delete` с промис-интерфейсом. Схема хранилищ и структура объектов приведены в **Приложении В**.

Выбор IndexedDB вместо localStorage обусловлен отсутствием ограничения на объём данных (localStorage: 5–10 МБ), поддержкой транзакций и возможностью хранения бинарных данных.

**localStorage.** Локальное хранилище браузера используется для двух целей: хранения CRDT-снапшотов (`Y.encodeStateAsUpdate` в base64) с ключом `vlab_ydoc_{roomId}` и хранения пользовательского профиля (`vlab_user`). Это обеспечивает персистентность состояния между перезагрузками страницы без необходимости обращения к серверу.

**Cache API и Service Worker.** Service Worker реализует кэширование статических ресурсов (HTML, JS, CSS) и WASM-бинарников по стратегии Cache First: при запросе ресурс сначала ищется в кэше, и только при отсутствии загружается из сети. Это обеспечивает офлайн-доступность приложения и WASM-движков после первой загрузки.

**SQLite на сервере.** На стороне сервера для хранения синхронизируемых курсов используется SQLite через Entity Framework Core. Такой выбор обусловлен лёгкостью развёртывания (single-file БД), пригодностью для умеренного количества курсов (десятки тысяч) и отсутствием необходимости в полноценном СУБД-сервере для рассматриваемого сценария.

## 2.4 Архитектура системы

### 2.4.1 Общая архитектура Local-First

Система построена по принципу Local-First: все пользовательские данные хранятся локально на устройстве, а сетевая инфраструктура служит средством синхронизации состояния между участниками. Общая схема развёртывания приведена на рисунке 2.1.

```
┌─────────────────────────────────────────────────────────┐
│                    Браузер пользователя                  │
│                                                         │
│  ┌──────────────┐    ┌──────────────┐  ┌─────────────┐ │
│  │   React UI   │    │   Y.Doc      │  │ Service     │ │
│  │  (Workspace) │◄───│  (CRDT)      │  │ Worker      │ │
│  └──────────────┘    └──────┬───────┘  │ (Cache API) │ │
│                             │          └─────────────┘ │
│  ┌──────────────┐    ┌──────▼───────┐                  │
│  │ WASM Engine  │    │  IndexedDB   │                  │
│  │  (in-browser)│    │  (Курсы,     │                  │
│  └──────────────┘    │   Сессии)    │                  │
│                       └─────────────┘                  │
└──────────────────────────┬──────────────────────────────┘
                           │ WebSocket (SignalR / MessagePack)
                ┌──────────▼──────────┐
                │  YARP Gateway       │
                └──────────┬──────────┘
                           │
                ┌──────────▼──────────┐
                │  ASP.NET Core 8     │
                │  ├─ SyncHub         │ ← CRDT relay (stateless)
                │  ├─ CourseSyncCtrl  │ ← REST: PUT /api/courses
                │  └─ ExecuteCtrl     │ ← REST: POST /api/execute
                │     (для бенчмарка) │
                └──────────┬──────────┘
                           │
                     ┌─────▼─────┐
                     │ SQLite    │ ← Хранение курсов
                     └───────────┘
```

Рисунок 2.1 – Общая архитектура Local-First платформы

Потоки данных:

1. Пользователь создаёт или импортирует курс. Данные сохраняются в IndexedDB; параллельно (при наличии сети) выполняется PUT-запрос к `/api/courses/{id}` для синхронизации между устройствами.
2. При открытии лабораторной работы создаётся `Y.Doc` с файлами лабы в `Y.Map('files')`.
3. Изменения кода через Monaco Editor транслируются в `Y.Text` через `MonacoBinding`.
4. Yjs-операции персистируются в localStorage через `Y.encodeStateAsUpdate` с base64-кодированием.
5. При наличии сети Yjs-операции отправляются через SignalR на сервер, ретранслируются остальным участникам комнаты и применяются на их `Y.Doc`.
6. Исполнение кода: UI Thread инициализирует WASM-движок (или использует уже инициализированный), выполняет код в изолированной WASM-песочнице, получает stdout/stderr.

### 2.4.2 Архитектура WASM-песочницы

Для каждого WASM-движка реализован единый интерфейс `Compiler`, описанный в листинге 2.1.

Листинг 2.1 – Интерфейс Compiler для WASM-движков

```typescript
interface Compiler {
  id: string                                      // 'python' | 'lua' | ...
  name: string                                    // 'Python (Pyodide WASM)'
  monacoLang: string                              // подсветка Monaco
  template: string                                // начальный шаблон
  isDownloaded(): Promise<boolean>                // проверка кэша
  downloadForOffline(p: (n: number) => void): Promise<void>
  init(): Promise<void>                           // ленивая инициализация VM
  run(files: Record<string, string>,
      logOutput: (line: string) => void,
      stdin?: string): Promise<void>
  checkForUpdates?(): Promise<boolean>
  removeOffline(): Promise<void>
}
```

Единый интерфейс позволяет компонентам `Workspace`, `useTestRunner` и `useProfiler` работать с любым компилятором без знания его внутренней реализации. Добавление нового языка сводится к созданию нового объекта, соответствующего `Compiler`, без изменения потребительского кода.

Изоляция WASM-исполнения в текущей версии обеспечивается двумя уровнями:

1. **WebAssembly-песочница на уровне браузера.** Каждый WASM-модуль выполняется в собственной линейной памяти, не имеет прямого доступа к DOM, файловой системе устройства, другим окнам и вкладкам. Это даёт изоляцию по умолчанию без дополнительных усилий разработчика.
2. **Тайм-аут на уровне UI Thread.** Запуск кода обёрнут в Promise с тайм-аутом 5 секунд. По истечении интервала исполнение прерывается. В будущей версии тайм-аутное прерывание планируется усилить через `Web Worker` с `worker.terminate()`, что позволит прерывать даже бесконечные циклы пользовательского кода в самой WASM-VM. На момент написания работы Pyodide и Wasmoon исполняются в основном потоке через изолированный экземпляр VM.

Архитектура исполнения для случая с Web Worker представлена на рисунке 2.2.

```
UI Thread
│
├── runCode() вызов
│
▼
Web Worker (изолированный поток)
│
├── Загрузка WASM-движка (при первом запуске)
│   └── Кэширование в Cache API
│
├── Получение файлов через postMessage
│   └── Запись в виртуальную ФС (Pyodide MEMFS)
│
├── Запуск кода
│
└── Тайм-аут: setTimeout(5000, () => worker.terminate())
    │
    ├── [Успех] postMessage(stdout) → UI
    └── [Тайм-аут] worker.terminate() → новый Worker при следующем запуске
```

Рисунок 2.2 – Архитектура WASM-песочницы с Web Worker и тайм-аутом

Такая архитектура обеспечивает:

- **изоляцию** — код студента не имеет доступа к DOM, объектам основного потока, другим вкладкам;
- **защиту от DoS** — бесконечный цикл (например, `while True: pass` в Python) прерывается через 5 секунд без зависания браузера;
- **поддержку офлайн-режима** — WASM-бинарники кэшируются в Cache API при первой загрузке.

Детальная диаграмма последовательности выполнения кода (включая альтернативный путь с тайм-аутом) приведена в **Приложении Б.4**.

### 2.4.3 Архитектура CRDT-синхронизации

Структура `Y.Doc` для одной лабораторной работы:

```
Y.Doc
├── Y.Map('files')       — файлы лабораторной работы
│   ├── 'main.py'   → Y.Text (содержимое файла)
│   ├── 'utils.py'  → Y.Text
│   └── 'input.txt' → Y.Text
└── Y.Map('presence')    — онлайн-статус участников
    ├── userId1 → { username, color, role, labRole? }
    └── userId2 → { username, color, role, labRole? }
```

Жизненный цикл `Y.Doc`:

1. **Инициализация.** При создании хука `useYjsSession` создаётся `Y.Doc`. Если в localStorage существует снапшот для данной комнаты, он восстанавливается через `Y.applyUpdate`. Если снапшот отсутствует и переданы `initialFiles` (шаблон лабораторной), `Y.Map` инициализируется содержимым шаблона в одной транзакции.

2. **Персистентность.** При каждом изменении `Y.Doc` срабатывает обработчик `'update'`, сохраняющий полное состояние в localStorage. Это обеспечивает восстановление состояния при перезагрузке страницы без серверного запроса.

3. **Подключение к комнате.** Если доступна сеть, создаётся SignalR-соединение; клиент вызывает `JoinRoom(roomId)`. Сервер уведомляет остальных участников через событие `UserJoined`.

4. **Late-joiner-протокол.** При получении события `UserJoined` каждый уже подключённый клиент отправляет полный снапшот своего `Y.Doc` через `SendDocumentUpdate`. Новый участник получает один или несколько снапшотов и применяет их через `Y.applyUpdate` — CRDT гарантирует идемпотентность (повторное применение одной операции не меняет результат).

5. **Инкрементальная синхронизация.** При каждом локальном изменении `Y.Doc` генерируется событие `'update'` с бинарной дельтой. Если дельта не помечена origin `'signalr'` (то есть не является входящей от другого участника), она отправляется через `SendDocumentUpdate` в SignalR Hub.

Диаграмма последовательности late-joiner-протокола с указанием всех передаваемых сообщений приведена в **Приложении Б.3**. Упрощённая схема представлена на рисунке 2.3.

    │                    │◄─────────────────────│
    │  UserJoined()      │                      │
    │◄───────────────────│                      │

### 2.4.4 Архитектура многофайлового Monaco Editor

Ключевая задача многофайлового редактора — обеспечить мгновенное переключение между файлами без пересоздания экземпляра Monaco Editor и без потери CRDT-биндинга.

Решение: один экземпляр редактора, множество моделей.

```
MultiFileEditor
│
├── editor instance (один на всё время жизни компонента)
│
├── modelsRef: Map<filePath, ITextModel>    ← кэш моделей
│   ├── 'main.py'   → ITextModel (+ MonacoBinding к Y.Text)
│   └── 'utils.py'  → ITextModel
│
└── activeBindingRef: MonacoBinding | null  ← активный биндинг
```

Рисунок 2.4 – Архитектура многофайлового Monaco Editor

Алгоритм переключения вкладки `filePath`:

1. Получить или создать `ITextModel` для `filePath`.
2. Если `MonacoBinding` для данного файла уже существует и активен — не пересоздавать.
3. Иначе уничтожить предыдущий `activeBindingRef.destroy()` и создать новый `MonacoBinding(ytext, model, editor)`.
4. Вызвать `editor.setModel(model)` — Monaco мгновенно переключает отображаемый документ без перерендеринга компонента.
5. Установить флаг read-only через `editor.updateOptions({ readOnly: true })`, если файл присутствует в списке `readOnlyFiles`.

Такая архитектура обеспечивает:

- сохранение истории undo/redo каждого файла отдельно;
- отсутствие мерцания при переключении вкладок;
- корректную CRDT-синхронизацию: каждый файл имеет независимый `Y.Text`.

Детальная диаграмма последовательности переключения вкладки (с созданием `ITextModel` и `MonacoBinding`) приведена в **Приложении Б.5**.

### 2.4.5 Архитектура синхронизации курсов

Курсы преподавателя синхронизируются между устройствами через REST API, минуя CRDT (поскольку редактирование курсов — однопользовательская операция в типичном сценарии). Архитектура включает:

- хук `useCourseStore` управляет загрузкой и сохранением курсов в IndexedDB;
- хук `useCourseSyncManager` периодически опрашивает сервер `GET /api/courses` и обновляет локальные курсы;
- при сохранении курса локально вызывается `PUT /api/courses/{id}` с заголовком `X-Device-Id` для идентификации устройства;
- сервер применяет стратегию Last-Write-Wins по полю `updatedAt`: если серверная версия новее, она возвращается клиенту, иначе клиентская версия записывается в БД.

Такая модель обеспечивает простоту реализации (плоский JSON-документ в SQLite) при достаточной для образовательного сценария согласованности.

## 2.5 Пользовательский интерфейс

### 2.5.1 Общие принципы

Интерфейс разработан в соответствии с концепцией IDE-like UX: тёмная тема, монохромная палитра, акцентный цвет — синий. Приоритет отдан функциональности над декоративностью: максимум пространства занимает редактор кода. Скриншоты всех реализованных экранов приведены в **Приложении Д**.

### 2.5.2 Описание экранов

**Главная страница (HomePage).** Отображает список доступных курсов в виде сетки карточек. При отсутствии курсов показывает пустое состояние с кнопкой импорта. Заголовок содержит кнопку «Импортировать курс» (выбор JSON-файла) и навигацию к профилю и настройкам.

**Страница курса (CoursePage).** Отображает список лабораторных работ курса. Каждая лабораторная представлена карточкой с названием, описанием и кнопкой «Открыть».

**Редактор курса (CourseEditorPage).** Позволяет преподавателю создавать и редактировать курс: добавлять/удалять лабораторные работы, управлять файлами каждой лабы (добавить файл, удалить, пометить как read-only, отредактировать содержимое), создавать тест-кейсы. Кнопка «Экспортировать» сохраняет курс в JSON-файл.

**Страница сессии (SessionPage / Workspace).** Основной экран работы обучающегося. Состоит из трёх колонок:

- левая (200 px): `FileTree` — дерево файлов с кнопкой добавления и удаления;
- центральная (flex): `MultiFileEditor` — Monaco Editor с вкладками открытых файлов;
- правая (320 px): `Terminal` (вывод программы, stdin, кнопка «Запустить») и `TestPanel` (если есть тест-кейсы).

Верхняя панель управления содержит: название лабы, код комнаты (кнопка копирования ссылки), выбор языка, кнопку описания задания, кнопку сброса к шаблону, кнопку профилировщика, список участников, статус загрузки движка.

**ProfilerPanel.** Модальное диалоговое окно. Пользователь задаёт количество итераций (по умолчанию 100), запускает бенчмарк. Поддерживает два режима: `wasm-only` (только клиентское измерение) и `compare` (одновременный замер серверного времени через `POST /api/execute`). Отображает: гистограмму распределения времени выполнения (20 интервалов), таблицу статистик (минимум, максимум, среднее, медиана, 95-й перцентиль для WASM и серверной оценки), кнопку экспорта Markdown-отчёта.

**Настройки (Settings).** Отображает информацию о кэшированных WASM-движках (Pyodide, Wasmoon, sql.js) с возможностью очистки кэша. Показывает состояние IndexedDB и localStorage.

**Профиль пользователя (ProfileSetupDialog).** Модальное окно первого запуска: пользователь вводит имя и выбирает цвет. Профиль сохраняется в localStorage и используется как идентификатор в presence-системе CRDT.
