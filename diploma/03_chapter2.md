# Глава 2. Проектирование системы

## 2.1 Технологии и инструменты разработки

На основании требований, сформулированных в Главе 1, выполнен выбор технологического стека. Ключевым критерием при выборе каждого инструмента являлась совместимость с Local-First архитектурой: приоритет отдавался решениям, способным функционировать без постоянного серверного соединения.

### 2.1.1 Фронтенд

**TypeScript + React (Vite)**

TypeScript — статически типизированное надмножество JavaScript. Использование статической типизации в данном проекте критически важно: интерфейсы типов `Course`, `Lab`, `TestCase`, `User`, `Participant` формируют контракт между клиентской и серверной частями, минимизируя класс ошибок несоответствия структур данных при сериализации/десериализации.

React выбран как доминирующий фреймворк для сложных интерактивных SPA с развитой экосистемой. Компонентная модель React хорошо соответствует архитектуре рабочего пространства: независимые компоненты `FileTree`, `MultiFileEditor`, `Terminal`, `TestPanel`, `ProfilerPanel` монтируются и размонтируются без взаимных зависимостей.

Vite выбран как инструмент сборки вместо Create React App по причине значительно более быстрого времени сборки (HMR < 100 мс) и нативной поддержки ES-модулей без дополнительной конфигурации.

**Monaco Editor**

Monaco Editor — движок редактора кода, лежащий в основе Visual Studio Code. Выбор обусловлен несколькими факторами. Во-первых, Monaco обеспечивает профессиональный UX редактирования кода: подсветка синтаксиса для 40+ языков, IntelliSense, мультикурсорное редактирование. Во-вторых, Monaco предоставляет низкоуровневый API управления моделями документов (`ITextModel`), что необходимо для реализации многофайловых вкладок без пересоздания экземпляра редактора. В-третьих, для Monaco существует официальный CRDT-биндинг `y-monaco`, реализующий двустороннюю синхронизацию между `Y.Text` и Monaco `ITextModel`.

Альтернатива — CodeMirror 6 — рассматривалась, однако экосистема y-codemirror менее зрелая по сравнению с y-monaco, а переносимость Monaco ITextModel удобнее для многофайловой архитектуры.

**Material UI (MUI v5)**

MUI — компонентная библиотека, реализующая Material Design. Использование готовой системы компонентов (`Box`, `Paper`, `Select`, `Chip`, `IconButton`, `Collapse`) позволило сосредоточиться на бизнес-логике приложения, а не на разработке UI-примитивов. Тёмная тема (`bgcolor: '#0a0a0a'`) настроена как глобальная для создания профессионального вида IDE.

**Yjs + y-monaco**

Yjs — CRDT-библиотека, обоснованная в разделе 1.2.3. Структура данных `Y.Map<filePath, Y.Text>` позволяет хранить все файлы лабораторной работы в одном `Y.Doc` с автоматической синхронизацией. `y-monaco` реализует `MonacoBinding` — объект, связывающий `Y.Text` с `ITextModel` и обрабатывающий трансляцию операций в обе стороны.

**@microsoft/signalr + @microsoft/signalr-protocol-msgpack**

Клиентская часть SignalR обеспечивает подключение к серверному хабу с автоматическим переподключением. MessagePack-протокол выбран вместо JSON по причине меньшего размера сообщений (бинарный формат): CRDT-дельты представляют собой массивы байт, которые в MessagePack передаются в нативном бинарном виде, тогда как JSON требует base64-кодирования.

### 2.1.2 WASM-движки

**Pyodide 0.25**

Pyodide — полная реализация CPython 3.11 в WebAssembly, разработанная Mozilla и переданная в Pyodide community. Включает менеджер пакетов micropip, поддерживающий установку Python-пакетов прямо в браузере. Для данной работы используется файловая система Emscripten (MEMFS) для монтирования многофайловых проектов: каждый файл лабораторной работы записывается в виртуальную ФС перед запуском скрипта.

**Fengari**

Fengari — реализация Lua 5.3 VM в JavaScript с WASM-ускорением. Выбрана за компактность (~200 КБ) и быструю инициализацию (~100 мс), что делает её удобной для учебных задач.

**sql.js**

sql.js — SQLite 3, скомпилированный в WebAssembly. Предоставляет полный SQL-интерфейс в браузере. База данных создаётся в памяти при каждом запуске, что соответствует модели «исполнение → вывод результата» без сохранения состояния между запусками.

### 2.1.3 Бэкенд

**C# ASP.NET Core 8 + SignalR**

Серверная часть реализована на C# ASP.NET Core 8. Выбор C# вместо Node.js или Java обусловлен зрелостью реализации SignalR в экосистеме .NET: библиотека разработана Microsoft, обеспечивает нативную интеграцию с ASP.NET Core middleware, поддерживает горизонтальное масштабирование через Redis Backplane.

SignalR Hub реализует минимальный набор операций: `JoinRoom`, `LeaveRoom`, `SendDocumentUpdate`. Сервер является stateless-ретранслятором: он не хранит содержимое кода и не интерпретирует CRDT-дельты — только пересылает их участникам комнаты.

**Docker Compose + nginx**

Развёртывание осуществляется через Docker Compose: контейнер nginx обрабатывает HTTP-запросы и проксирует WebSocket-соединения SignalR на ASP.NET Core контейнер. Nginx также отдаёт статические файлы фронтенда (собранный Vite bundle).

### 2.1.4 Хранение данных

**IndexedDB**

IndexedDB — основное хранилище данных на клиентской стороне. Используется для хранения курсов (`courses` store) и метаданных сессий (`sessions` store). Доступ реализован через собственный типизированный wrapper на базе нативного IDB API, предоставляющий функции `get<T>`, `getAll<T>`, `save`, `delete` с промис-интерфейсом. Схема хранилищ и структура объектов приведены в **Приложении В**.

Выбор IndexedDB вместо localStorage обусловлен: отсутствием ограничения на объём данных (localStorage: 5–10 МБ), поддержкой транзакций, возможностью хранения бинарных данных.

**localStorage**

localStorage используется для хранения CRDT-снапшотов (`Y.encodeStateAsUpdate` → base64) и пользовательского профиля. Ключ снапшота: `vlab_ydoc_{roomId}`. Это обеспечивает персистентность состояния между перезагрузками страницы без необходимости обращения к серверу.

**Cache API + Service Worker**

Service Worker реализует кэширование статических ресурсов (HTML, JS, CSS) и WASM-бинарников по стратегии Cache First: при запросе ресурс сначала ищется в кэше, и только при отсутствии загружается из сети. Это обеспечивает офлайн-доступность приложения и WASM-движков после первой загрузки.

## 2.4 Архитектура системы

### 2.4.1 Общая архитектура Local-First

Система построена по принципу Local-First: все данные хранятся локально на устройстве пользователя, а сетевая инфраструктура служит исключительно для синхронизации состояния между участниками.

```
┌─────────────────────────────────────────────────────────┐
│                    Браузер студента                      │
│                                                         │
│  ┌──────────────┐    ┌──────────────┐  ┌─────────────┐ │
│  │   React UI   │    │   Y.Doc      │  │ Service     │ │
│  │  (Workspace) │◄───│  (CRDT)      │  │ Worker      │ │
│  └──────────────┘    └──────┬───────┘  │ (Cache API) │ │
│                             │          └─────────────┘ │
│  ┌──────────────┐    ┌──────▼───────┐                  │
│  │  Web Worker  │    │  IndexedDB   │                  │
│  │ (WASM Engine)│    │  (Курсы,     │                  │
│  └──────────────┘    │   Сессии)    │                  │
│                       └─────────────┘                  │
└──────────────────────────┬──────────────────────────────┘
                           │ WebSocket (SignalR)
                           │ CRDT-дельты (MessagePack)
                ┌──────────▼──────────┐
                │  ASP.NET Core       │
                │  SignalR Hub        │
                │  (Ретранслятор)     │
                └─────────────────────┘
```

**Потоки данных:**

1. Пользователь создаёт или импортирует курс → данные сохраняются в IndexedDB.
2. При открытии лабораторной работы создаётся `Y.Doc` с файлами лабы в `Y.Map('files')`.
3. Изменения кода через Monaco Editor → MonacoBinding → операции Yjs → `Y.Text`.
4. Yjs-операции персистируются в localStorage (`Y.encodeStateAsUpdate` → base64).
5. При наличии сети: Yjs-операции отправляются через SignalR на сервер → ретранслируются остальным участникам комнаты → применяются на их `Y.Doc`.
6. Исполнение кода: UI Thread → `postMessage` → Web Worker → WASM Engine → результат в stdout → `postMessage` обратно.

### 2.4.2 Архитектура WASM-песочницы

```
UI Thread
│
├── runCode() вызов
│
▼
Web Worker (изолированный поток)
│
├── Загрузка WASM-движка (при первом запуске)
│   └── Кэширование в Cache API
│
├── Получение файлов через postMessage
│   └── Запись в виртуальную ФС (Pyodide MEMFS)
│
├── Запуск кода
│
└── Watchdog: setTimeout(5000, () => worker.terminate())
    │
    ├── [Успех] postMessage(stdout, stderr) → UI
    └── [Таймаут] worker.terminate() → новый Worker при следующем запуске
```

Каждый WASM-компилятор реализует единый интерфейс:

```typescript
interface Compiler {
  id: string
  name: string
  isDownloaded(): Promise<boolean>
  downloadForOffline(progress: (p: number) => void): Promise<void>
  init(): Promise<void>
  run(files: Record<string, string>, logOutput: (line: string) => void, stdin: string): Promise<void>
}
```

Такая архитектура обеспечивает:
- **Изоляцию:** код студента не имеет доступа к DOM, объектам основного потока, другим вкладкам.
- **Защиту от DoS:** бесконечный цикл (`while True: pass`) прерывается через 5 секунд без зависания браузера.
- **Поддержку офлайн:** WASM-бинарники кэшируются в Cache API при первой загрузке.

Детальная диаграмма последовательности выполнения кода (включая альтернативный путь с таймаутом) приведена в **Приложении Б.4**.

### 2.4.3 Архитектура CRDT-синхронизации

**Структура Y.Doc:**

```
Y.Doc
├── Y.Map('files')       — файлы лабораторной работы
│   ├── 'main.py'   → Y.Text (содержимое файла)
│   ├── 'utils.py'  → Y.Text
│   └── 'input.txt' → Y.Text
└── Y.Map('presence')    — онлайн-статус участников
    ├── userId1 → { username, color, role }
    └── userId2 → { username, color, role }
```

**Жизненный цикл Y.Doc:**

1. **Инициализация:** при создании хука `useYjsSession` создаётся `Y.Doc`. Если в localStorage существует снапшот для данной комнаты — он восстанавливается через `Y.applyUpdate`. Если снапшот отсутствует и переданы `initialFiles` (шаблон лабораторной) — `Y.Map` инициализируется содержимым шаблона.

2. **Персистентность:** при каждом изменении `Y.Doc` срабатывает обработчик `'update'`, сохраняющий полное состояние в localStorage. Это обеспечивает восстановление состояния при перезагрузке страницы без серверного запроса.

3. **Подключение к комнате:** если доступна сеть, создаётся SignalR-соединение; клиент вызывает `JoinRoom(roomId)`. Сервер уведомляет остальных участников через событие `UserJoined`.

4. **Late-joiner протокол:** при получении события `UserJoined` каждый уже подключённый клиент отправляет полный снапшот своего Y.Doc через `SendDocumentUpdate`. Новый участник получает один или несколько снапшотов и применяет их через `Y.applyUpdate` — CRDT гарантирует идемпотентность (повторное применение одной операции не меняет результат).

5. **Инкрементальная синхронизация:** при каждом локальном изменении Y.Doc генерируется событие `'update'` с бинарной дельтой. Если дельта не помечена origin `'signalr'` (т.е. не является входящей от другого участника), она отправляется через `SendDocumentUpdate` в SignalR Hub.

Диаграмма последовательности late-joiner протокола с указанием всех передаваемых сообщений приведена в **Приложении Б.3**. Упрощённая схема:

```
Участник A          SignalR Hub          Участник B (новый)
    │                    │  JoinRoom(roomId)     │
    │                    │◄─────────────────────│
    │  UserJoined()      │                      │
    │◄───────────────────│                      │
    │ SendDocumentUpdate(fullSnap)              │
    │───────────────────►│──────────────────────►│ Y.applyUpdate(snap)
```

### 2.4.4 Архитектура многофайлового Monaco Editor

Ключевая задача многофайлового редактора — обеспечить мгновенное переключение между файлами без пересоздания экземпляра Monaco Editor и без потери CRDT-биндинга.

**Решение: один экземпляр редактора, множество моделей.**

```
MultiFileEditor
│
├── editor instance (один на всё время жизни компонента)
│
├── modelsRef: Map<filePath, ITextModel>    ← кэш моделей
│   ├── 'main.py'   → ITextModel (+ MonacoBinding к Y.Text)
│   └── 'utils.py'  → ITextModel
│
└── activeBindingRef: MonacoBinding | null  ← активный биндинг
```

**Переключение вкладки (filePath):**
1. Получить или создать `ITextModel` для `filePath`.
2. Если `MonacoBinding` для данного файла уже существует и активен — не пересоздавать.
3. Иначе: уничтожить предыдущий `activeBindingRef.destroy()`, создать новый `MonacoBinding(ytext, model, editor)`.
4. Вызвать `editor.setModel(model)` — Monaco мгновенно переключает отображаемый документ без перерендеринга компонента.

Такая архитектура обеспечивает:
- Сохранение истории undo/redo каждого файла отдельно.
- Отсутствие мерцания при переключении вкладок.
- Корректную CRDT-синхронизацию: каждый файл имеет независимый `Y.Text`.

Детальная диаграмма последовательности переключения вкладки (с созданием ITextModel и MonacoBinding) приведена в **Приложении Б.5**.

## 2.5 Пользовательский интерфейс

### 2.5.1 Общие принципы

Интерфейс разработан в соответствии с концепцией IDE-like UX: тёмная тема, монохромная палитра (#0a0a0a, #141414, #1e1e1e), акцентный цвет — синий (#2196f3). Приоритет отдан функциональности над декоративностью: максимум пространства занимает редактор кода. Скриншоты всех реализованных экранов приведены в **Приложении Д**.

### 2.5.2 Описание экранов

**Главная страница (HomePage)**

Отображает список доступных курсов в виде сетки карточек. При отсутствии курсов показывает пустое состояние с кнопкой импорта. Заголовок содержит кнопку «Импортировать курс» (выбор JSON-файла) и навигацию к профилю и настройкам.

**Страница курса (CoursePage)**

Отображает список лабораторных работ курса с прогрессом. Каждая лабораторная представлена карточкой с названием, описанием и кнопкой «Открыть».

**Редактор курса (CourseEditorPage)**

Позволяет преподавателю создавать и редактировать курс: добавлять/удалять лабораторные работы, управлять файлами каждой лабы (добавить файл, удалить, пометить как read-only, отредактировать содержимое), создавать тест-кейсы. Кнопка «Экспортировать» сохраняет курс в JSON-файл.

**Рабочая область (Workspace)**

Основной экран работы студента. Состоит из трёх колонок:
- Левая (200px): FileTree — дерево файлов с кнопкой добавления и удаления.
- Центральная (flex): MultiFileEditor — Monaco Editor с вкладками открытых файлов.
- Правая (320px): Terminal (вывод программы, stdin, кнопка Run) + TestPanel (если есть тест-кейсы).

Верхняя панель управления содержит: название лабы, код комнаты (кнопка копирования ссылки), выбор языка, кнопку описания задания, кнопку сброса к шаблону, кнопку профилировщика, список участников, статус загрузки движка.

**ProfilerPanel**

Модальное диалоговое окно. Пользователь задаёт количество итераций (по умолчанию 100), запускает бенчмарк. Отображает: гистограмму распределения времени выполнения (20 бинов), таблицу статистик (min, max, avg, median, p95 для WASM и оценку для сервера), кнопку экспорта Markdown-отчёта.

**Настройки (SettingsPage)**

Отображает информацию о кэшированных WASM-движках (Pyodide, Fengari, sql.js) с возможностью очистки кэша. Показывает состояние IndexedDB и localStorage.
