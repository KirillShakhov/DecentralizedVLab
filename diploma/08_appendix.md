Рисунок Б.1 – Диаграмма вариантов использования

Акторы:
- Преподаватель — создаёт курсы, добавляет лабораторные работы и тест-кейсы, экспортирует/импортирует курсы, настраивает read-only-файлы.
- Обучающийся — открывает и выполняет лабораторные работы, редактирует код, запускает тесты и профилировщик, участвует в совместных сессиях.

Примечание. Роли не требуют аутентификации в текущей реализации — профиль пользователя создаётся локально при первом запуске и хранится в localStorage.

### В.2 Хранение CRDT-снапшотов (localStorage)

CRDT-снапшоты `Y.Doc` не хранятся в IndexedDB, а сохраняются в localStorage по соображениям синхронности API:

```
localStorage
├── vlab_ydoc_{roomId}     string  (base64(Y.encodeStateAsUpdate(ydoc)))
├── vlab_selected_compiler string  ('python' | 'lua' | ...)
└── vlab_user              string  (JSON: { id, username, color, createdAt })
```

Ключ `vlab_ydoc_{roomId}` обновляется при каждом изменении `Y.Doc` через обработчик события `'update'`. При следующем открытии той же комнаты состояние восстанавливается через `Y.applyUpdate`.

### В.3 Серверная СУБД (SQLite, EF Core)

Для синхронизации курсов между устройствами на сервере используется SQLite.

```

│                    Table: Courses                             │
│                    Primary Key: Id                            │
├──────────────────────────────────────────────────────────────┤
│  Id            TEXT     PK   (UUID v4)                       │
│  UpdatedAt     INTEGER       (Unix timestamp ms)             │
│  JsonData      TEXT          (полный JSON курса)             │
│  LastDeviceId  TEXT          (X-Device-Id отправителя)       │
└──────────────────────────────────────────────────────────────┘
```

Рисунок В.2 – Схема серверного хранилища курсов

Хранение «JSON-документ + временная метка» (а не нормализация по таблицам) выбрано исходя из следующих соображений: курс является целостной единицей синхронизации (не редактируется по частям), плоское хранение упрощает миграции при изменении схемы, а объём курсов остаётся в пределах сотен килобайт даже для крупных учебных дисциплин.

## Приложение Д

**Скриншоты реализованного приложения**

Скриншоты выполненного приложения прилагаются в отдельных файлах изображений. Ниже приводятся описания каждого экрана.

### Д.1 Главная страница (HomePage)

Отображает сетку курсов. В заголовке — кнопка импорта JSON-файла курса. При отсутствии курсов показывается пустое состояние с предложением импортировать первый курс. В правом верхнем углу — иконка профиля пользователя и переход к настройкам.

### Д.2 Страница курса (CoursePage)

Отображает список лабораторных работ выбранного курса в виде карточек. Каждая карточка содержит название, описание и кнопку «Открыть». Сверху расположена кнопка «Создать сессию», открывающая модальное окно выбора лабораторной для совместной работы.

### Д.3 Редактор курса (CourseEditorPage)

Список лабораторных работ в виде раскрывающихся панелей (Accordion). Каждая лабораторная содержит: поля названия и описания, список файлов с редактором содержимого и переключателем read-only, список тест-кейсов с полями stdin и ожидаемого вывода, опциональный список ролей для совместных лабораторных. Кнопки «Экспортировать» (сохранение в JSON) и «Сохранить» в заголовке.

### Д.4 Рабочая область (Workspace)

Три колонки: FileTree (200 px) | Monaco Editor (flex) | Terminal + TestPanel (320 px). Верхняя панель: название лабораторной, код комнаты с кнопкой копирования ссылки, выбор языка, кнопка описания задания, кнопка сброса к шаблону, кнопка профилировщика, список участников, статус движка.

### Д.5 Многофайловый редактор

Вкладки открытых файлов с иконками по расширению (🐍 .py, 🟨 .js, 🌙 .lua, 🗄️ .sql, ☕ .java). Активная вкладка подчёркнута синей линией. Кнопка закрытия появляется при наведении.

### Д.6 Панель тестирования (TestPanel)

Список тест-кейсов со статусами: ⏳ pending → 🔄 running → ✅ pass / ❌ fail. Прогресс-бар и счётчик пройденных тестов. При раскрытии провалившегося теста — сравнение фактического и ожидаемого вывода в режиме diff.

### Д.7 Профилировщик (ProfilerPanel)

Модальное диалоговое окно. Настройка числа итераций, переключатель режимов «WASM only» и «Сравнение с сервером». После завершения отображается 20-интервальная гистограмма распределения, таблица статистик (минимум, среднее, медиана, 95-й перцентиль, σ, RTT), сравнительная строка для серверной оценки (в режиме сравнения), кнопка «Экспортировать отчёт» в формате Markdown.

### Д.8 Список участников (ParticipantList)

Цветные аватары-маркеры с именами участников текущей сессии в верхней панели Workspace. Обновляются в реальном времени через `Y.Map('presence')`. При нажатии на аватар отображаются дополнительные данные участника (роль, цвет, идентификатор).

### Д.9 Страница настроек (Settings)

Информация о кэшированных WASM-движках (Pyodide, Wasmoon, sql.js) с размером и датой кэширования. Кнопки очистки кэша для каждого движка и для всего Cache Storage. Информация о состоянии IndexedDB (количество курсов и сессий, общий объём).

### Д.10 Диалог выбора роли (RolePickerDialog)

Модальное окно при подключении к совместной лабораторной с заданными ролями. Отображает список ролей с описаниями и закреплёнными за ними файлами. После выбора роль фиксируется через `Y.Map('presence')` и определяет, какие файлы доступны участнику для редактирования.

### Д.11 Диалог настройки профиля (ProfileSetupDialog)

Модальное окно первого запуска. Поле ввода имени пользователя и палитра выбора цвета. Профиль сохраняется в localStorage и используется для идентификации в presence-системе CRDT и при синхронизации курсов через заголовок `X-Device-Id`.
