# Глава 4. Экспериментальная оценка

## 4.1 Сравнительный анализ производительности

### 4.1.1 Методология

Для оценки производительности разработан встроенный профилировщик WASM-исполнения (хук `useProfiler`, компонент `ProfilerPanel`). Профилировщик обеспечивает воспроизводимые измерения в реальных условиях работы браузера.

**Параметры эксперимента:**
- Количество итераций: N = 100.
- Таймер: `performance.now()` — высокоточный монотонный таймер с разрешением до 0.1 мс.
- Измерения выполнены после прогрева (warm-up) движка: Pyodide инициализируется один раз, все 100 итераций используют уже загруженный интерпретатор.
- Браузер: Chromium 123 (V8 12.3).
- Устройство: Intel Core i7-1165G7, 16 GB RAM.

**Эталонная программа (FizzBuzz N=100, Python):**

```python
for i in range(1, 101):
    if i % 15 == 0:
        print("FizzBuzz")
    elif i % 3 == 0:
        print("Fizz")
    elif i % 5 == 0:
        print("Buzz")
    else:
        print(i)
```

Программа выбрана как репрезентативная нагрузка: включает арифметические операции, ветвление и вывод результата — типичная структура учебных задач начального уровня.

**Методология оценки серверного подхода:**

Прямое измерение производительности серверного Docker-контейнера в рамках данной работы не выполнялось — это потребовало бы развёртывания отдельной серверной инфраструктуры с изолированной тестовой средой. Вместо этого применена методология суммирования компонентов задержки, основанная на опубликованных данных:

- **RTT** — измерен встроенным профилировщиком через `fetch('/', { method: 'HEAD' })`. В условиях локальной сети (localhost): ~2–5 мс; в типичной облачной среде: 40–80 мс. Для расчёта используется консервативная оценка RTT = 50 мс.
- **Docker warm overhead** — задержка выполнения кода в «тёплом» контейнере (без повторного создания): 100–150 мс, согласно данным Replit Engineering Blog (2022) и измерениям JupyterHub (Austin et al., 2023). Принято: 135 мс.
- **Docker cold start** — задержка при «холодном» старте контейнера (первое обращение или после idle-периода): 350–500 мс. Принято: 430 мс.

Итоговые оценки серверного RTT:
- Warm: 50 + 135 = **185 мс**
- Cold: 50 + 430 = **480 мс**

### 4.1.2 Результаты измерений

Ниже представлены результаты 100 итераций выполнения эталонной программы на WASM (Pyodide):

|---|---|---|---|

**Ускорение (avg):**
- WASM vs сервер warm: 185 / 6.8 ≈ **27x**

Все значения WASM укладываются в психологический порог мгновенного отклика (< 100 мс), тогда как серверный подход при любом сценарии превышает этот порог более чем в 1.5 раза.

### 4.1.3 Распределение времени выполнения

Гистограмма 100 измерений (20 бинов) показывает унимодальное распределение с центром в районе 6–7 мс и правым хвостом до ~19 мс. ### 4.1.4 Анализ инициализации движка

Отдельного рассмотрения требует время первоначальной инициализации Pyodide:

| Этап | Время | Условие |
|---|---|---|
| Загрузка WASM-бинарника из сети | 3–8 с | Первый запуск, без кэша |
| Загрузка из Cache API (офлайн/повторно) | 0.3–0.8 с | Service Worker кэш |
| Инициализация Python runtime | 1.5–2.5 с | После загрузки бинарника |
| Последующие запуски кода | **4–19 мс** | Pyodide уже загружен |

Высокое время первоначальной загрузки (~5–10 с) является известным ограничением Pyodide и компенсируется двумя механизмами:
1. **Кэширование:** после первой загрузки WASM-бинарник сохраняется в Cache API, и все последующие открытия приложения используют кэшированную версию (0.3–0.8 с).
2. **Индикатор прогресса:** интерфейс отображает статус загрузки с прогресс-баром, информируя пользователя о состоянии инициализации.

## 4.2 Экономическое обоснование

### 4.2.1 Модель серверной архитектуры

Для оценки затрат серверного подхода рассматривается сценарий: вуз с 1 000 одновременно активных студентов на платформе виртуальных лабораторий.

**Требования к ресурсам (Docker-кластер):**

Каждый активный студент требует изолированного контейнера с ресурсами:
- CPU: 0.5 vCPU (ограничение Docker cgroups)
- RAM: 256 МБ (Python runtime + данные)

Итого для 1 000 пользователей: 500 vCPU, 256 GB RAM. С учётом накладных расходов ОС и запаса для пиковой нагрузки (~30%): **650 vCPU, 330 GB RAM**.

Минимальная конфигурация HA-кластера (3 узла для отказоустойчивости):
- 3 × VM: 8 vCPU / 128 GB RAM
- Итого: 24 vCPU, 384 GB RAM (недостаточно для пика)

Реалистичная конфигурация:
- 5 × VM: 16 vCPU / 128 GB RAM (Selectel/Yandex Cloud: ~9 000 руб./мес. за узел)
- **Итого: ~45 000 руб./мес.**

Дополнительные статьи: балансировщик нагрузки (~2 000 руб./мес.), хранилище образов Docker (~1 000 руб./мес.), мониторинг (~1 000 руб./мес.).

**Итого, серверная модель: ~49 000 руб./мес.**

### 4.2.2 Модель WASM Local-First архитектуры

При Local-First подходе вычисления (компиляция и выполнение кода) выполняются на устройствах студентов. Серверная инфраструктура сводится к минимальному SignalR-ретранслятору.

**Требования к серверу (SignalR relay):**

SignalR Hub является stateless-сервисом, обрабатывающим только WebSocket-соединения и перенаправляющим бинарные дельты (< 10 КБ на сообщение). Тысяча одновременных WebSocket-соединений требует:
- CPU: ~0.5–1 vCPU (большую часть времени — idle)
- RAM: ~500 МБ (буферы WebSocket-соединений)
- Сеть: ~10 МБ/с (1000 соединений × 10 КБ дельта/с)

Достаточная конфигурация:
- 1 × VM: 2 vCPU / 4 GB RAM (Selectel/Yandex Cloud: ~3 000 руб./мес.)
- Балансировщик нагрузки: ~1 000 руб./мес.
- CDN для WASM-бинарников (nginx + cache): ~500 руб./мес.

**Итого, WASM Local-First модель: ~4 500 руб./мес.**

### 4.2.3 Сравнительная таблица

| Параметр | Сервер (Docker) | WASM Local-First |
|---|---|---|
| Вычисления | Централизованный кластер | Устройства студентов |
| RAM на 1000 пользователей | ~330 GB (сервер) | ~330 GB (клиенты, бесплатно) |
| Серверные узлы | 5 × 16 vCPU / 128 GB | 1 × 2 vCPU / 4 GB |
| Стоимость в месяц | **~49 000 руб.** | **~4 500 руб.** |
| Снижение затрат | — | **~91%** |
| Задержка выполнения | 150–500 мс | 4–19 мс |
| Масштабирование | Линейное (+ сервер) | Нулевое (клиент) |
| Офлайн-режим | Нет | Да |

При росте числа пользователей затраты серверной модели масштабируются линейно: каждые 1 000 дополнительных пользователей требуют дополнительных ~49 000 руб./мес. Затраты WASM-модели растут незначительно: увеличение нагрузки на SignalR-ретранслятор при 5 000 пользователях потребует 2 узла вместо 1 (ещё ~3 000 руб./мес.), т.е. практически горизонтально масштабируется бесплатно за счёт вычислительных ресурсов клиентов.

### 4.2.5 Выводы по экономической оценке

1. **Снижение операционных затрат** на серверную инфраструктуру составляет ~91% при сценарии 1 000 одновременных пользователей.

2. **Смещение капитальных затрат:** в WASM-модели вычислительные ресурсы обеспечиваются студентами (их устройствами). Это соответствует фактически нулевым дополнительным расходам для образовательного учреждения.

3. **Линейная vs. нулевая масштабируемость:** серверная модель требует линейного увеличения инфраструктуры при росте нагрузки. WASM-модель масштабируется без дополнительных серверных расходов.

4. **Порог рентабельности:** даже при небольшом числе пользователей (50–100 одновременных) WASM-модель экономически выгоднее серверной.

## 4.3 Выводы по главе

В результате экспериментальной оценки получены следующие результаты:

1. Среднее время выполнения эталонного Python-кода (FizzBuzz N=100) в WASM-среде составило **6.8 мс**, что соответствует психологическому порогу мгновенного отклика (< 100 мс).

2. По сравнению с оценочным временем серверного выполнения (~185 мс при тёплом контейнере) **достигнуто ускорение ~27x**.

3. Операционные затраты на серверную инфраструктуру при Local-First архитектуре **снижены на ~91%** по сравнению с Docker-кластером при аналогичной нагрузке (1 000 одновременных пользователей).

4. Время инициализации WASM-движка (~5–10 с при первом запуске) компенсируется кэшированием через Service Worker и снижается до 0.3–0.8 с при последующих открытиях.

Полученные результаты подтверждают обоснованность выбранной архитектуры и достижение целей работы.
