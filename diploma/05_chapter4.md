# 4 ЭКСПЕРИМЕНТАЛЬНАЯ ОЦЕНКА

## 4.1 Сравнительный анализ производительности

### 4.1.1 Методология

#### Цель эксперимента

Целью является эмпирическое подтверждение гипотезы: клиентское исполнение кода в WASM-среде обеспечивает значительно меньшую задержку по сравнению с серверным исполнением в Docker-контейнере и укладывается в психологический порог мгновенного отклика (менее 100 мс согласно [2]).

#### Измерительный инструмент

Все измерения выполнены с помощью встроенного профилировщика системы (хук `useProfiler`). Для измерения времени используется `performance.now()` — высокоточный монотонный таймер Web API с разрешением до 0.1 мс, не подверженный коррекциям системных часов. Таймер запускается непосредственно перед вызовом `compiler.run()` и останавливается по завершении промиса:

```typescript
const t0 = performance.now()
await compiler.run(files, () => {}, stdin)
const elapsed = performance.now() - t0
```

Каждое измерение включает: подготовку файлов для запуска, инициализацию виртуальной файловой системы WASM-движка (если требуется), собственно интерпретацию кода и получение вывода в UI Thread. Между итерациями каждые 10 шагов выполняется вынужденный возврат управления через `setTimeout(0)` для предотвращения блокировки UI.

#### Параметры эксперимента

- **Количество итераций:** N = 200 (повышено по сравнению с предварительными измерениями для снижения стандартной ошибки).
- **Прогрев (warm-up):** первые 10 итераций исключаются из статистики. Это устраняет эффект JIT-компиляции браузера и первоначальной инициализации структур данных WASM-движка.
- **Браузеры:** Chromium 123 (V8 12.3), Firefox 124 (SpiderMonkey), Safari 17 (JavaScriptCore + WASM).
- **Тестовое устройство:** ноутбук Intel Core i7-1165G7 (4 ядра, 2.8 GHz), 16 GB RAM, ОС Windows 11. Измерения выполнены при закрытых фоновых процессах и выключенном фоновом сетевом трафике.

#### Тестовые задачи

Выбраны четыре задачи с различной вычислительной характеристикой, типичные для учебных курсов программирования:

**Задача Т1: FizzBuzz (лёгкая, O(N), ветвление + вывод)**
```python
for i in range(1, 101):
    if i % 15 == 0: print("FizzBuzz")
    elif i % 3 == 0: print("Fizz")
    elif i % 5 == 0: print("Buzz")
    else: print(i)
```
Характеристика: арифметические операции, условные переходы, строковый вывод. Типична для вводного курса.

**Задача Т2: Сортировка пузырьком (средняя, O(N²), работа с массивом)**
```python
import random
arr = list(range(1000, 0, -1))
for i in range(len(arr)):
    for j in range(len(arr) - i - 1):
        if arr[j] > arr[j+1]:
            arr[j], arr[j+1] = arr[j+1], arr[j]
print(arr[0], arr[-1])
```
Характеристика: 500 000 итераций внутреннего цикла, интенсивная работа с памятью. Типична для курса алгоритмов.

**Задача Т3: Монте-Карло π (тяжёлая, вещественная арифметика)**
```python
import random, math
n, inside = 100000, 0
for _ in range(n):
    x, y = random.random(), random.random()
    if math.sqrt(x*x + y*y) <= 1.0:
        inside += 1
print(4 * inside / n)
```
Характеристика: 100 000 итераций с вещественной арифметикой и вызовами `math.sqrt`. Нагрузка на FPU и RNG движка.

**Задача Т4: Рекурсивный Фибоначчи (средняя, рекурсия)**
```python
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)
print(fib(30))
```
Характеристика: ~2.7 млн рекурсивных вызовов, нагрузка на call stack. Fib(30) — стандартная учебная задача, не приводящая к переполнению стека в CPython.

#### Методология оценки серверного подхода

**Важное замечание:** прямое измерение производительности промышленного Docker-контейнера в рамках данной работы не выполнялось — это требует развёртывания изолированного стенда с управляемой нагрузкой, что выходит за рамки поставленных задач. Серверное время представляет собой **расчётную оценку**, основанную на трёх измеримых компонентах:

| Компонент | Значение | Источник |
|---|---|---|
| RTT (измерен) | 50 мс | `fetch HEAD /`, среднее по 100 замерам |
| Docker warm overhead | 135 мс | Replit Engineering Blog, 2022; источник [1] |
| Docker cold start | 430 мс | Там же, оценка при простое более 5 минут |
| **Итого warm** | **185 мс** | Сумма |
| **Итого cold** | **480 мс** | RTT + cold |

Оценка является **нижней границей** реального серверного времени: в неё не включены задержки очереди планировщика задач при пиковой нагрузке и overhead контейнерной сети.

**Анализ чувствительности к RTT:** таблица ниже показывает, как ускорение изменяется при других значениях RTT для задачи Т1:

| RTT (мс) | Сервер warm (мс) | Ускорение WASM |
|---|---|---|
| 20 (локальная сеть) | 155 | ~22x |
| 50 (типичный) | 185 | ~27x |
| 80 (пиковый) | 215 | ~31x |
| 120 (нестабильный) | 255 | ~37x |

Вывод: даже при наилучшем для сервера сценарии (RTT = 20 мс, тёплый контейнер) WASM-подход обеспечивает ускорение >20x.

### 4.1.4 Измерение потребления памяти

Полные данные по потреблению памяти приведены в **Приложении Г.3**. Потребление памяти Web Worker процессом измерялось через `window.performance.memory.usedJSHeapSize` (Chromium) до и после выполнения каждой задачи:

| Задача | Δ RAM (МБ) | Комментарий |
|---|---|---|
| Т1 FizzBuzz | +0.2 | Минимальный вывод |
| Т2 BubbleSort | +1.8 | Массив 1000 элементов в heap |
| Т3 Monte-Carlo | +0.9 | Только скалярные переменные |
| Т4 Fibonacci | +0.4 | Стек вызовов — освобождается по завершении |

Накладные расходы памяти самого Pyodide (базовый Python runtime в WASM) составляют ~40–60 МБ в Web Worker и не включены в Δ RAM выше. Это значение постоянно на протяжении всех итераций — Pyodide не пересоздаётся между запусками.

При серверном подходе каждый Docker-контейнер потребляет ~256 МБ RAM (Python runtime + данные + overhead контейнера). При 1 000 одновременных пользователей это составляет ~256 ГБ суммарно на сервере, тогда как при WASM подходе каждый браузер потребляет ~60 МБ локально, не создавая нагрузки на серверную инфраструктуру.

### 4.1.6 Угрозы достоверности

Академическая строгость требует явного обозначения ограничений проведённого эксперимента.

**Внутренняя достоверность (Internal Validity):**
- Все измерения выполнены на одном устройстве → исключена межмашинная вариабельность, но результаты отражают только данное оборудование.
- Не исключён эффект термального троттлинга CPU при длинных серийных измерениях (задача Т3): рост температуры процессора может снижать частоту к концу серии. Для нейтрализации каждая задача измерялась в отдельном сеансе с паузой 30 с.
- GC паузы V8 включены в измерения `performance.now()` — они завышают P95 и max. Медиана устойчива к GC и является предпочтительной метрикой для оценки типичной задержки.

**Внешняя достоверность (External Validity):**
- Тестовые задачи (Т1–Т4) соответствуют учебному уровню начальных и средних курсов программирования. Задачи с NumPy (матричные вычисления) или Pandas (обработка датафреймов) могут показать иное соотношение WASM и нативного исполнения, поскольку Pyodide использует WASM-порты C-библиотек, чья производительность зависит от SIMD и других расширений [8].
- Результаты получены на десктопном CPU. На мобильных устройствах (ARM Cortex-A, ограниченная TDP) ожидаются значения в 2–4 раза выше, однако общая тенденция (WASM значительно быстрее серверного RTT) сохраняется.

**Достоверность конструкта (Construct Validity):**
- `performance.now()` измеряет wall-clock time, а не CPU time. При высокой загрузке браузерного процесса другими задачами (рендеринг, JS таймеры) результаты могут быть завышены.
- Серверное время представляет расчётную оценку, а не прямое измерение — это является принципиальным ограничением. Полная экспериментальная верификация потребует развёртывания контрольного Docker-стенда в той же сетевой топологии.

## 4.2 Оценка ресурсоёмкости центральной инфраструктуры

Анализ чувствительности к ключевым допущениям приведён в **Приложении Г.5**.

Цель данного раздела — количественно оценить вычислительную нагрузку на центральную серверную инфраструктуру при двух архитектурных подходах: классическом серверном (Docker-контейнеры на стороне сервера для каждого активного пользователя) и предложенном Local-First (исполнение кода на устройстве пользователя, сервер выступает только как ретранслятор CRDT-дельт). Финансовая оценка не входит в задачи работы; используются только аппаратные показатели — vCPU, оперативная память, пропускная способность сети.

### 4.2.1 Сценарии и допущения

Анализ выполнен для трёх типичных сценариев образовательного учреждения, представленных в таблице 4.5.

Таблица 4.5 – Сценарии нагрузки

| Сценарий | Одновременных пользователей | Описание |
|---|---|---|
| Малый вуз / курс | 100 | Одна-две учебных группы |
| Средний вуз | 1000 | Несколько факультетов |
| Крупный вуз | 5000 | Университет с онлайн-курсами |

**Допущения для серверной модели:**

- пиковая нагрузка составляет 100 % одновременных пользователей (консервативная оценка);
- каждый изолированный контейнер исполнения кода — 0,5 vCPU и 256 МБ RAM (типичный лимит для Python-интерпретатора);
- развёртывание в режиме высокой доступности предполагает дублирование узлов;
- ресурсы провижионируются с запасом 30 % для пиковых всплесков.

**Допущения для WASM Local-First модели:**

- вычисления выполняются на устройствах пользователей; нагрузка на серверный CPU за исполнение кода равна нулю;
- сервер выполняет только функцию SignalR-ретранслятора CRDT-дельт и обслуживания REST API курсов;
- для хранения и раздачи статических WASM-бинарников достаточно одного узла на любое реалистичное число пользователей.

| 30 % (оптимистично) | ~58 | 2 | в 29 раз |

Даже при самом оптимистичном сценарии нагрузки серверная модель требует на порядок больше центральных вычислительных ресурсов, чем Local-First.

**Прочие операционные накладные расходы.** Помимо аппаратных ресурсов, серверная модель требует постоянного администрирования кластера контейнеров (мониторинг, алертинг, отказоустойчивость, обновления безопасности). Local-First модель снимает этот класс задач: серверная часть сводится к одному ретранслятору, не имеющему доступа к пользовательскому коду и не требующему таких же мер изоляции, как многопользовательский исполнитель.

## 4.3 Выводы по главе

В результате экспериментальной оценки получены следующие результаты:

1. **Производительность WASM.** Для всех четырёх тестовых задач (FizzBuzz, BubbleSort, Monte-Carlo π, Fibonacci) среднее время выполнения в Chromium составило 6.8 мс, 24.6 мс, 58.1 мс и 15.7 мс соответственно. Все значения укладываются в порог мгновенного отклика (< 100 мс).

2. **Ускорение.** По сравнению с расчётным временем серверного warm-выполнения ускорение составляет от 8x (Monte-Carlo, сложная вещественная арифметика) до 27x (FizzBuzz, типичная учебная задача).

3. **Кросс-браузерность.** WASM подход работает во всех трёх основных браузерах; наихудший результат (Safari, 11.2 мс для FizzBuzz) в 16 раз лучше серверного warm-выполнения.

4. **Потребление памяти.** Накладные расходы памяти на стороне сервера (~256 МБ/пользователь) полностью переносятся на клиентское устройство (~60 МБ Pyodide runtime), снимая нагрузку с серверной инфраструктуры.

5. **Ресурсоёмкость центральной инфраструктуры.** При 1000 одновременных пользователях серверная архитектура требует около 195 vCPU и 330 ГБ RAM на стороне центральных серверов, тогда как Local-First архитектура — 2 vCPU и 4 ГБ RAM (ретранслятор CRDT). Это соответствует снижению потребления центральных вычислительных ресурсов в 80–95 раз.

6. **Чувствительность.** Даже при самых консервативных допущениях (RTT = 20 мс, тёплый контейнер, нагрузка 30 %) WASM Local-First подход обеспечивает ускорение более 22× и кратное снижение нагрузки на центральную инфраструктуру.

Полученные результаты подтверждают достижение цели работы: повышение скорости отклика и отказоустойчивости среды выполнения учебного кода и снижение нагрузки на центральную инфраструктуру за счёт Local-First архитектуры на базе WebAssembly.
