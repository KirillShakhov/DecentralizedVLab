# ЗАКЛЮЧЕНИЕ

В ходе выполнения выпускной квалификационной работы проведено исследование технологий разработки виртуальных лабораторий для технических предметных областей и реализована децентрализованная Local-First платформа виртуальных лабораторий на базе WebAssembly и CRDT.

## Итоги по задачам

**Задача 1. Анализ существующих платформ и технологий.** Проведён сравнительный анализ шести платформ виртуальных лабораторий (Stepik, Replit, JupyterHub, CodeSandbox, PhET, Pyodide). Выявлено, что все серверные платформы имеют принципиальные ограничения: задержка выполнения 150–500 мс, линейный рост аппаратной ресурсоёмкости центральной инфраструктуры, отсутствие офлайн-режима. Выполнено сравнение технологий исполнения кода в браузере (Skulpt, Pyodide, Wasmoon, sql.js) и подходов синхронизации (OT и CRDT), обоснован выбор WASM и Yjs.

**Задача 2. Проектирование Local-First PWA-архитектуры.** Спроектирована четырёхкомпонентная архитектура: WASM-песочница (с тайм-аутным контролем времени, целевая схема — Web Worker с принудительным завершением), CRDT-синхронизация (`Y.Map<filePath, Y.Text>` и SignalR-ретранслятор), IndexedDB как первичное хранилище курсов, Service Worker для офлайн-кэширования. Дополнительно спроектирована синхронизация курсов через REST API с применением стратегии Last-Write-Wins. Разработаны схемы данных и диаграммы взаимодействия компонентов.

**Задача 3. Разработка WASM-песочницы.** Реализованы четыре WASM-компилятора (Python/Pyodide, JavaScript/V8, Lua/Wasmoon, SQLite/sql.js), соответствующие единому интерфейсу `Compiler` (полный исходный код — **Приложение А**). Обеспечена изоляция через WebAssembly-песочницу браузера и контроль времени через тайм-аут на уровне UI Thread. Реализовано кэширование WASM-бинарников через Cache API для офлайн-работы.

**Задача 4. Реализация CRDT-модуля совместного редактирования.** Реализован хук `useYjsSession`, инкапсулирующий: инициализацию `Y.Doc`, восстановление состояния из localStorage, late-joiner-протокол, SignalR-синхронизацию с фильтрацией эхо-обновлений, файловые операции и presence-механизм. Реализован многофайловый Monaco Editor с ленивым созданием `ITextModel` и `MonacoBinding`, а также с разделением прав на запись через флаг read-only.

**Задача 5. Экспериментальная оценка и анализ ресурсоёмкости.** Разработан встроенный профилировщик (`useProfiler`, `ProfilerPanel`) с N-итерационным измерением через `performance.now()`, статистиками (минимум, максимум, среднее, медиана, 95-й перцентиль), гистограммой и экспортом Markdown-отчёта. Поддержан режим прямого сравнения с серверным временем через `POST /api/execute`. Получены измеренные результаты WASM-производительности и проведена количественная оценка снижения нагрузки на центральную инфраструктуру с анализом чувствительности к ключевым допущениям.

## Достигнутые результаты

В результате выполнения работы получены следующие измеримые результаты:

- **Ускорение выполнения кода.** Среднее время выполнения Python-кода в WASM-среде составило 6,8 мс для эталонной задачи FizzBuzz, что соответствует ускорению около 27× по сравнению с расчётным серверным временем (около 185 мс при тёплом контейнере). Для всех четырёх тестовых задач (FizzBuzz, BubbleSort, Monte-Carlo π, Fibonacci) WASM-время укладывается в порог мгновенного отклика 100 мс [2].
- **Снижение нагрузки на центральную инфраструктуру.** При 1000 одновременных пользователях серверная архитектура требует около 195 vCPU и 330 ГБ RAM на стороне центральных серверов, тогда как Local-First архитектура — 2 vCPU и 4 ГБ RAM (ретранслятор CRDT). Это соответствует снижению потребления центральных вычислительных ресурсов в 80–95 раз. Даже при оптимистичном для сервера сценарии (нагрузка 30 %) Local-First подход требует на порядок меньше центральных вычислительных ресурсов.
- **Офлайн-режим.** Приложение функционирует при отсутствии интернет-соединения после первоначального кэширования благодаря Service Worker (стратегия Cache First) и сохранению состояния в IndexedDB и localStorage.
- **Совместная работа.** Реализовано одновременное редактирование кода несколькими участниками с конечной согласованностью (Yjs CRDT, алгоритм YATA), включая late-joiner-протокол и presence-механизм с цветовой идентификацией.
- **Кросс-браузерность.** Решение протестировано в браузерах Chromium 123, Firefox 124, Safari 17. Во всех браузерах достигается значение времени выполнения ниже психологического порога 100 мс.

## Выводы о соответствии цели и задачам

Цель работы — повышение скорости отклика и отказоустойчивости среды выполнения учебного кода за счёт разработки Local-First платформы виртуальных лабораторий — достигнута. Получено ускорение в 8–27 раз в зависимости от характера задачи и снижение нагрузки на центральную вычислительную инфраструктуру на один-два порядка при сохранении функциональной полноты, требуемой для образовательного применения.

Все пять задач, сформулированных во введении и техническом задании, выполнены в полном объёме. Положения, выносимые на защиту, подтверждены экспериментальными измерениями и архитектурным анализом.

## Направления дальнейшего развития

1. **Перевод исполнения в Web Worker.** В текущей версии WASM-движки исполняются в основном UI-потоке. Перенос исполнения в Web Worker с принудительным `worker.terminate()` обеспечит полное прерывание блокирующих операций пользовательского кода (бесконечных циклов) без воздействия на отзывчивость интерфейса.

2. **Расширение поддержки языков.** Добавление Java через CheerpJ или TeaVM (компиляция Java в WASM/JS), C/C++ через Emscripten, R через WebR, Rust через wasm-pack.

3. **LTI-интеграция.** Реализация протокола LTI 1.3 (Learning Tools Interoperability) для интеграции платформы с существующими LMS (Moodle, Canvas, Sakai), что позволит использовать её как внешний инструмент без замены инфраструктуры вуза.

4. **P2P-режим через WebRTC.** При наличии прямой видимости между участниками (локальная сеть) возможен переход от SignalR relay к прямому P2P-соединению через WebRTC (`y-webrtc`), что полностью устранит серверную зависимость для совместной работы.

5. **Удалённые курсоры (Remote Cursors).** Интеграция курсорного awareness `y-monaco` для отображения позиции курсора и выделения каждого участника в реальном времени.

6. **Аналитика прогресса.** Хранение результатов выполнения тест-кейсов в IndexedDB с агрегацией и визуализацией прогресса обучающегося по курсу. Опциональная синхронизация с сервером для просмотра преподавателем.

7. **Скрытые тест-кейсы.** Поле `isHidden` в типе `TestCase` уже присутствует в модели данных. Требуется реализация UX, скрывающего детали скрытых тест-кейсов от обучающегося до момента финальной сдачи лабораторной работы.

8. **Сжатие CRDT-снапшотов.** Применение `Y.snapshot` для периодической компактификации `Y.Doc`, накапливающего tombstone-записи при интенсивном редактировании.

9. **Полноценная серверная изоляция.** При необходимости предоставления серверного исполнения недоверенному коду (например, для проверки сложных задач, не помещающихся в браузерную WASM-память) — интеграция Docker-в-Docker, gVisor или Firecracker.

10. **Аутентификация.** Текущая реализация работает с локальным профилем без серверной аутентификации. Для production-развёртывания в учебном заведении потребуется интеграция с университетскими SSO-провайдерами (OAuth 2.0, SAML).

Полученное решение демонстрирует жизнеспособность Local-First подхода как основы для нового поколения образовательных платформ программирования, обеспечивающего одновременно высокую отзывчивость, низкую ресурсоёмкость центральной инфраструктуры, отказоустойчивость и поддержку совместной работы.
