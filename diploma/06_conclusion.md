# ЗАКЛЮЧЕНИЕ

В ходе выполнения выпускной квалификационной работы проведено исследование технологий разработки виртуальных лабораторий для технических предметных областей и реализована децентрализованная Local-First платформа виртуальных лабораторий на базе WebAssembly и CRDT.

## Итоги по задачам

**Задача 1. Анализ существующих платформ и технологий.**

Проведён сравнительный анализ шести платформ виртуальных лабораторий (Stepik, Replit, JupyterHub, CodeSandbox, PhET, Pyodide). Выявлено, что все серверные платформы имеют принципиальные ограничения: задержка выполнения 150–500 мс, линейный рост стоимости инфраструктуры, отсутствие офлайн-режима. Выполнено сравнение технологий исполнения кода в браузере (Skulpt, Pyodide, Fengari, sql.js) и подходов синхронизации (OT vs CRDT), обоснован выбор WASM и Yjs.

**Задача 2. Проектирование Local-First PWA архитектуры.**

Спроектирована четырёхкомпонентная архитектура: WASM-песочница (Web Worker + таймаут), CRDT-синхронизация (`Y.Map<filePath, Y.Text>` + SignalR relay), IndexedDB как первичное хранилище, Service Worker для офлайн-кэширования. Разработаны схемы данных и диаграммы взаимодействия компонентов.

**Задача 3. Разработка WASM-песочницы.**

Реализованы четыре WASM-компилятора (Python/Pyodide, JavaScript/V8, Lua/Fengari, SQLite/sql.js), соответствующие единому интерфейсу `Compiler` (полный исходный код — **Приложение А**). Обеспечена изоляция через Web Worker и защита от DoS через таймаут 5 с (`worker.terminate()`). Реализовано кэширование WASM-бинарников через Cache API для офлайн-работы.

**Задача 4. Реализация CRDT-модуля совместного редактирования.**

Реализован хук `useYjsSession`, инкапсулирующий: инициализацию Y.Doc, восстановление состояния из localStorage, late-joiner протокол, SignalR-синхронизацию с фильтрацией эхо-обновлений, файловые операции. Реализован многофайловый Monaco Editor с ленивым созданием ITextModel и MonacoBinding.

**Задача 5. Экспериментальная оценка и экономическое обоснование.**

Разработан встроенный профилировщик (`useProfiler`, `ProfilerPanel`) с N-итерационным измерением через `performance.now()`, статистиками (min, max, avg, median, p95), гистограммой и экспортом Markdown. Получены измеренные результаты WASM-производительности и проведён расчёт экономической эффективности.

## Достигнутые результаты

В результате выполнения работы получены следующие измеримые результаты:

- **Ускорение выполнения кода:** среднее время выполнения Python-кода в WASM-среде составило 6.8 мс, что соответствует ускорению **~27x** по сравнению с серверным исполнением (~185 мс).
- **Снижение затрат на инфраструктуру:** операционные расходы при Local-First архитектуре составляют ~4 500 руб./мес. против ~49 000 руб./мес. для Docker-кластера при 1 000 одновременных пользователях — **снижение ~91%**.
- **Офлайн-режим:** приложение функционирует при отсутствии интернет-соединения после первоначального кэширования.
- **Совместная работа:** реализовано одновременное редактирование кода несколькими участниками с конечной согласованностью (Yjs CRDT).

## Направления дальнейшего развития

1. **Расширение поддержки языков:** добавление Java через CheerpJ (JVM в WASM), C/C++ через Emscripten, R через WebR.

2. **LTI-интеграция:** реализация протокола LTI 1.3 для интеграции платформы с существующими LMS (Moodle, Canvas), что позволит использовать её как внешний инструмент без замены инфраструктуры вуза.

3. **P2P-режим через WebRTC:** при наличии прямой видимости между участниками (локальная сеть) возможен переход от SignalR relay к прямому P2P-соединению через WebRTC (y-webrtc), что полностью устранит серверную зависимость для совместной работы.

4. **Аналитика прогресса:** хранение результатов выполнения тест-кейсов в IndexedDB с агрегацией и визуализацией прогресса студента по курсу.

5. **Скрытые тест-кейсы:** разделение тест-кейсов на открытые (видны студенту) и скрытые (проверяются только при финальной сдаче), аналогично Codeforces.
